## 上下文

当前 `SlaveView.vue` 的寄存器表格只有读取和修改运行时值的行内输入框，没有添加、编辑、删除寄存器定义的交互入口。后端已有 `add_register` 和 `remove_register` Tauri Commands，但前端从未调用。

前端技术栈为 Vue 3（Composition API）+ TypeScript，无 UI 组件库，使用 scoped CSS。

## 目标 / 非目标

**目标：**
- 在寄存器表格上方添加"添加寄存器"按钮
- 点击后弹出添加寄存器 Modal 对话框
- 表格行支持右键菜单，提供"编辑"和"删除"选项
- 添加/编辑共用同一 Modal，根据模式预填或清空字段
- 地址冲突时弹出确认提示（覆盖/取消）
- 高级选项（数据类型、字节序）默认折叠
- 所有 UI 文案中文

**非目标：**
- 不修改后端 Commands（已有实现）
- 不支持多选批量操作
- 不引入 UI 组件库（纯 CSS）
- 不修改寄存器运行时值的行内编辑（已存在）

## 决策

### D1: 添加/编辑共用一个 Modal

**选择**：同一 `RegisterModal.vue` 组件，通过 `mode: 'add' | 'edit'` prop 区分行为

**理由**：
- 代码复用，减少维护成本
- 两种场景的 UI 几乎相同，只有预填值和标题不同
- 避免维护两套相似代码

### D2: 右键菜单实现方式

**选择**：原生 CSS 绝对定位的浮动菜单，不引入第三方库

**理由**：
- 简单直接，无需额外依赖
- Vue 原生支持 `@contextmenu` 事件
- 菜单项少（仅 2 项），样式可控

### D3: 地址冲突检测时机

**选择**：在用户点击"确认"时检测，通过后端返回的确认信息或前端预检查实现

**具体方式**：
- 前端在提交前遍历当前寄存器列表，检查同类型同地址是否已存在
- 若存在，弹出确认框询问用户"覆盖"或"取消"

**理由**：减少不必要的网络请求，提升用户体验。

### D4: 高级选项折叠

**选择**：前端控制显示/隐藏状态，使用 `v-show` 切换

**理由**：纯前端状态，不涉及后端通信，切换无性能开销。

## 风险 / 权衡

- **[风险]** 右键菜单与浏览器默认右键冲突
  → 使用 `@contextmenu.prevent` 阻止默认行为，只显示自定义菜单
- **[风险]** Modal 中输入框获得焦点时右键仍会弹出系统菜单
  → 在 input 上也绑定 `@contextmenu.prevent`

## 开放问题

- 是否需要支持在添加时同时填写初始值（runtime value）？
  → 暂不支持，保持简单，后续可扩展
