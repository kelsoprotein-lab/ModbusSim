## 上下文

RegisterTable.vue 是从站模式的核心交互组件，展示选中从站设备的所有寄存器定义及其当前值。ValuePanel.vue 是右侧值解析面板，将选中寄存器的原始 u16 值解析为多种数据格式。

当前问题：
- `<table>` 元素直接设 `overflow-y: auto` 无效，导致超出视口的行不可见
- 没有搜索/过滤输入，只有按功能码类型的过滤
- 只支持单选，ValuePanel 自动读取连续地址做 32/64 位解析，用户无法控制

## 目标 / 非目标

**目标：**
- 修复表格滚动，使 100+ 寄存器可完整浏览
- 提供地址/名称搜索，快速定位目标寄存器
- 支持 Ctrl/Shift 多选寄存器，手动控制参与 32/64 位组合解析的寄存器
- 键盘上下箭头导航

**非目标：**
- 不做虚拟滚动（当前最大 101 行/类型，性能不是瓶颈）
- 不改变后端 API
- 不增加拖拽排序
- 不增加列排序功能

## 决策

### 1. 滚动修复方案

**选择**：将 `<table>` 包裹在 `<div class="table-scroll-container">` 中，容器设 `overflow-y: auto; flex: 1`，`<thead>` 保持 `position: sticky; top: 0`。

**替代方案**：使用 CSS `display: block` 在 `<tbody>` 上实现滚动。

**理由**：包裹 div 方案更简单可靠，不会破坏表格的列宽对齐。`display: block` 方案需要手动对齐 thead 和 tbody 的列宽。

### 2. 搜索框实现

**选择**：在 `table-header-bar` 中增加 `<input>` 搜索框，绑定 `searchQuery` ref。`filteredRegisters` 在现有类型过滤之后叠加搜索过滤：如果输入以 `0x` 开头则按十六进制地址匹配，纯数字按十进制地址匹配，否则按名称模糊匹配（包含即可，不区分大小写）。

**替代方案**：分开的地址输入和名称输入。

**理由**：单一搜索框更简洁，通过输入格式自动识别搜索意图，减少 UI 复杂度。

### 3. 多选模型

**选择**：
- `selectedRows` 从 `ref<Register | null>` 改为 `ref<Register[]>`
- 普通点击：替换选中（清空旧选中，选中当前行）
- Ctrl+点击：切换当前行的选中状态（不影响其他已选行）
- Shift+点击：从上一个点击的行到当前行的范围选中
- App.vue 中 `selectedRegister` 类型变为数组

**替代方案**：复选框列。

**理由**：Ctrl/Shift 多选是表格类应用的标准交互模式，不需要额外的 UI 列，更紧凑。

### 4. ValuePanel 多选解析

**选择**：
- 接收 `selectedRegister` 数组
- 1 个寄存器：显示 16 位解析（现有行为）
- 2 个寄存器：显示 16 位 + 32 位解析，使用选中的两个寄存器值（按地址排序）
- 4 个寄存器：显示 16 位 + 32 位 + 64 位解析
- 其他数量：只显示各自的 16 位解析
- 不再自动读取邻居寄存器，完全由用户选择决定

**替代方案**：保留自动读取，多选时覆盖。

**理由**：完全由用户选择更可预测，避免"自动读取的值和用户选中的值混在一起"的混淆。

### 5. 键盘导航

**选择**：在表格滚动容器上监听 `@keydown`，处理 `ArrowUp` / `ArrowDown`。移动选中行时自动滚动到可视区域（`scrollIntoView`）。

**理由**：简单直接，标准表格导航体验。

## 风险 / 权衡

- **[Breaking Change]** `selectedRegister` 类型从对象变为数组 → App.vue 和 ValuePanel.vue 需同步修改，但都是内部组件
- **[多选+搜索交互]** 搜索过滤后选中状态可能包含不在可见列表中的行 → 搜索时清空选中状态，避免混淆
